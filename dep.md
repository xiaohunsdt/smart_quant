## 2. 核心 Alpha 策略：双因子流动性真空模型

### 2.1 因子定义

1. **成交动能 (Tick Delta Velocity)**  
   滑动窗口（如 100ms）内主动买单量 - 主动卖单量的加速度：
   \[
   a_{\Delta}(t) = \frac{d^2}{dt^2}\left(\sum V_{buy} - \sum V_{sell}\right)
   \]

2. **撤单速率 (Cancellation Rate Velocity)**  
   监测 L1（最优买卖档位）的 `MDUpdateAction=2`（删除）报文的瞬时频率。  
   定义单位时间（如 10ms）内撤单数量 \(c(t)\)，其异常值为：
   \[
   z_c(t) = \frac{c(t) - \mu_c(t)}{\sigma_c(t)}
   \]
   其中 \(\mu_c,\sigma_c\) 为过去 1 秒滚动均值和标准差。

> [修正] 增加**动态阈值**：撤单速率不再使用固定 \(N\times\sigma\)，而是基于**时段分位数**（例如过去 30 秒的 95 分位数）。纽约开盘时段的分位数阈值远高于亚洲午盘。

### 2.2 信号触发逻辑

\[
\text{Signal} = (a_{\Delta}(t) > \text{Thresh}_{delta}) \ \cap\ (z_c(t) > \text{Thresh}_{cancel})
\]

- **预测窗口**：1.0s ~ 5.0s（放宽以适应真实网络+处理延迟 5-10ms）
- **逻辑解释**：成交爆发（真实吃单）与撤单爆发（虚假支撑消失）同时发生，确认流动性即将枯竭，价格将向买/卖方向跳空。

> [修正] 明确撤单因子的**方向性**：买单撤单速率激增利空（Bid 塌陷），卖单撤单速率激增利多（Ask 塌陷）。

## 3. 系统架构与 C++ 实现

### 3.1 极速工程优化（维持原设计）

- **零分配内存池**：启动时预分配全量结构体，运行期禁用 `new/malloc`。
- **绑核与隔离**：行情处理 + 下单逻辑绑定一个物理核（`isolcpus`）；可选单独线程负责 FIX 会话心跳，通过无锁队列传递信号。
- **FIX 报文预烘焙**：预分配缓冲区，触发时仅覆盖时间戳和 ClOrdID 后发送。

### 3.2 模块划分

| 模块 | 职责 |
|------|------|
| **Market Data Gateway** | 解析 `35=X` 增量刷新，维护 L1/L2 订单簿镜像，计算 Delta 与撤单速率 |
| **Alpha Logic** | 双因子实时计算、阈值比较、生成交易信号（买入/卖出/无） |
| **Order Manager** | FIX 会话管理（登录、心跳、序列号），发送 `35=D` (IOC) 及 `35=F` 撤单 |
| **Risk Manager** | 持仓计时、最大亏损熔断、冷静期控制 |
| **Binary Logger** | `mmap` 记录原始 FIX 流，用于回测 |

> [修正] **移除做市模式**：初期版本不实现双向限价挂单。黄金波动剧烈，做市模式风险过高且与 HFT 抢单逻辑干扰。后期若需加入，应单独分支并增加敞口限制。

## 4. 交易执行与风险控制

### 4.1 订单执行：IOC 模式

- **进场订单**：`OrdType=F` (Market) + `TimeInForce=IOC` (Tag 59=3)。  
  不允许使用 Limit 订单排队，因为流动性枯竭窗口内排队等于死等撤单。
- **出场**：
  - **止盈**：成交确认（`35=8`）后，立即在期望跳空价位挂出 **Limit IOC 平仓单**（若未立即成交则取消，避免过夜）。
  - **止损/超时**：持仓超过 **5 秒** 未平仓，发送 Market 订单强制平仓。

### 4.2 滑点模型与冷静期

系统实时计算**有效滑点** = |成交价 - 信号发出时的 L1 价格|。

若**连续 3 笔交易**滑点 > 2 ticks（或单笔滑点 > 5 ticks），自动进入 **60 秒冷静期**，停止交易并告警。

> [修正] 增加 **IOC 穿透滑点预估**：回测时必须模拟 “吃第一档 -> 若剩余量不足则吃第二档 -> …” 的逐档成交逻辑，而非简单的第一档成交。

## 5. 回测与验证体系（分三阶段）

### 5.1 数据采集要求

- 必须记录 `MDUpdateAction` (0=New, 1=Change, 2=Delete) 以及 `MDEntryType` (0=Bid, 1=Offer, 2=Trade)。
- 时间戳使用本地 TSC 纳秒时钟，对齐所有事件。

### 5.2 第一阶段：离线双因子有效性验证

- **目标**：确认双因子模型是否比单因子（仅 Delta）有显著提升。
- **指标**：
  1. 信号发出后 1~5 秒内价格向预测方向移动 ≥1 tick 的概率（胜率）。
  2. 对比单因子胜率 \(p_1\) 与双因子胜率 \(p_2\)，要求 \(p_2 - p_1 \ge 3\%\)（绝对提升）。
- **若未达标**：重新调整因子权重、窗口长度或阈值分位数，或放弃该策略。

### 5.3 第二阶段：影子测试（Shadow Trading）

- **环境**：Beeks LD5，连接实盘行情，不发送订单。
- **模拟执行**：在每个信号触发时，记录当前 L1 价格，等待 `Processing_latency(50μs) + Network_latency(200μs)` 后，用**逐档填充模型**模拟 IOC 订单的成交价与成交量。
- **输出**：预期胜率、平均滑点、理论日盈亏。

### 5.4 第三阶段：0.01 手实盘探测（Pilot）

- **目的**：获取真实 `Order-to-Ack` 延迟和物理排队位置导致的额外滑点。
- **监控指标**：
  - `Tick-to-Trade Latency`：从收到最后一个行情 tick 到发出 `35=D` 的时间。
  - `Order-to-Ack Latency`：发出订单到收到 `35=8` (ExecType=Fill) 的往返 RTT。
  - `Realized Slippage`：实盘成交价与信号发出时的 L1 价格之差。
- **决策门**：若连续 100 笔交易后，**净利润为负** 或 **中位数滑点 > 1.5 tick**，终止项目。
